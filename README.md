### Инструкция по выполнению домашнего задания

   1. Сделайте `fork` данного репозитория к себе в Github и переименуйте его по названию или номеру занятия, например, https://github.com/имя-вашего-репозитория/git-hw или  https://github.com/имя-вашего-репозитория/7-1-ansible-hw).
   2. Выполните клонирование данного репозитория к себе на ПК с помощью команды `git clone`.
   3. Выполните домашнее задание и заполните у себя локально этот файл README.md:
      - впишите вверху название занятия и вашу фамилию и имя
      - в каждом задании добавьте решение в требуемом виде (текст/код/скриншоты/ссылка)
      - для корректного добавления скриншотов воспользуйтесь [инструкцией "Как вставить скриншот в шаблон с решением](https://github.com/netology-code/sys-pattern-homework/blob/main/screen-instruction.md)
      - при оформлении используйте возможности языка разметки md (коротко об этом можно посмотреть в [инструкции  по MarkDown](https://github.com/netology-code/sys-pattern-homework/blob/main/md-instruction.md))
   4. После завершения работы над домашним заданием сделайте коммит (`git commit -m "comment"`) и отправьте его на Github (`git push origin`);
   5. В личном кабинете прикрепите и отправьте ссылку на решение в виде md-файла в вашем Github.
   6. Любые вопросы по выполнению заданий спрашивайте в разделе “Вопросы по заданию” в личном кабинете.
   
Желаем успехов в выполнении домашнего задания!

### Задание 2

**1.** Создаём файл `.gitignore` и проверяем статус
![](img/img-git-8-03-hw/create-gitignor.png)
В разделе Untracked files (Неотслеживаемые файлы) красным цветом выделено имя нового файла .gitignore. Git увидел, что появился новый элемент, но пока за ним не следит.

**2.** Добавление файла .gitignore в следующий коммит git add .gitignore
![](img/img-git-8-03-hw/add-gitignor.png)
Git начал отслеживать этот файл на локальном компьютере и добавил его в индекс (staged area)

**3.** Добавляем в файл .gitignore правила, чтобы игнорировать любые файлы .pyc, а также все файлы в директории cache.
![](img/img-git-8-03-hw/edit-gitignor01.png)
- echo *.pyc > .gitignore ... — запись правил игнорирования в файл.
- type .gitignore — показывает, что файл содержит две нужные строки: *.pyc и cache/.

Файл .gitignore сейчас находится одновременно в двух состояниях:
- Раздел Changes to be committed (Зелёная зона): new file: .gitignore. Это то состояние файла, которое ты зафиксировал командой git add, когда файл был еще пустым. Git запомнил его пустым и готов закоммитить в таком виде.
- Раздел Changes not staged for commit (Красная зона): modified: .gitignore. После того, как был сделан git add, файл был изменён (добавил туда правила *.pyc и cache/). Git видит эту разницу между «тем, что добавлено в индекс» и «тем, что реально сейчас лежит в папке».

**4.** Коммит и пуш.
![](img/img-git-8-03-hw/gitignore-commit-push.png)
Результат коммита
- `(git commit ...)[main 7d87f77]` — создан твой новый коммит на ветке main с уникальным хэшем 7d87f77.
- `1 file changed, 1 insertion(+)` — Git зафиксировал появление одного файла с твоими правилами.
- `create mode 100644 .gitignore` — эта строчка подтверждает, что Git создал в своей базе новый отслеживаемый файл .gitignore со стандартными правами доступа.

Результат пуша
- `(git push origin main)Writing objects: 100% (3/3), 301 bytes` — изменения весом в 301 байт успешно улетели на сервера GitHub.
- `b403099..7d87f77 main -> main` — удаленный репозиторий обновился. История на GitHub передвинулась с твоего прошлого коммита b403099 вперед на новый коммит 7d87f77. Теперь локальная папка и сайт снова полностью синхронизированы.

[Ссылка на коммит 7d87f77](https://github.com/murman051/git-homework/commit/7d87f770ab23cce0183f821fa9fc53b55245ed1e)

![](img/img-git-8-03-hw/3/step3.png)
- `git add test.sh` - добавляем новый файл в индекс
- `git commit -m "Create test.sh with initial feature script"` - делаем первый коммит для этого файла
- `git push -u origin dev` - отправляем новую ветку на GitHub

Новый файл test.sh успешно добавлен в индекс, закоммичен и отправлен в новую удалённую ветку dev на GitHub с использованием флага связи веток -u.

![](img/img-git-8-03-hw/3/step3-2.png)
- `echo echo "Fixing bugs in dev branch" >> test.sh` - дописываем вторую строчку в файл
- `git add test.sh` - снова добавляем изменённый файл в индекс
- `git commit -m "Update test.sh with second feature line"` - делаем второй коммит
- `git push origin dev` - делаем обычный пуш

В файл test.sh добавлена новая строка кода, изменения закоммичены и отправлены в удалённую ветку dev на GitHub. Таким образом сымитирована история из нескольких коммитов в ветке разработки.

**4.** Переключаемся обратно на главную ветку.
![](img/img-git-8-03-hw/3/step4.png)

- `git push origin main` - Пушим в основную ветку на GitHub

В основной ветке main успешно создан файл main.sh. Изменения закоммичены и запушены на GitHub. Теперь истории веток main и dev официально разошлись в стороны.

- `Your branch is up to date with 'origin/main'.` — локальная и удалённая ветки main полностью синхронизированы.
